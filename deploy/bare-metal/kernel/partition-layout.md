# MILNET SSO System — Disk Partition Layout

**CLASSIFICATION:** UNCLASSIFIED // FOUO

## Overview

The MILNET SSO appliance uses a four-partition layout designed for
cryptographic integrity verification and data-at-rest encryption.
The root filesystem is immutable (dm-verity), and all mutable data
resides on a LUKS2-encrypted partition with TPM2 auto-unlock.

```
+-----------------------------------------------------------------------+
|  GPT Partition Table                                                  |
|                                                                       |
|  /dev/sda1   EFI System Partition     512 MB    FAT32                |
|  /dev/sda2   Root Filesystem           5 GB     ext4   (dm-verity)   |
|  /dev/sda3   Verity Hash Tree        256 MB     raw    (hash data)   |
|  /dev/sda4   Encrypted Data         remaining   LUKS2  (ext4 inside) |
|                                                                       |
+-----------------------------------------------------------------------+
```

## Partition Details

### /dev/sda1 — EFI System Partition (ESP)

| Property     | Value                          |
|-------------|--------------------------------|
| Size        | 512 MB                         |
| Filesystem  | FAT32                          |
| GPT Type    | EFI System (C12A7328-...)      |
| Mount Point | /boot/efi                      |
| Flags       | boot, esp                      |

**Contents:**

```
/EFI/BOOT/BOOTX64.EFI          — Shim bootloader (signed by Microsoft UEFI CA)
/EFI/milnet/milnet-kernel.efi   — Signed MILNET kernel (MOK-signed EFI stub)
/EFI/milnet/milnet-initramfs.img — Initramfs (embedded in signed kernel or separate)
/loader/loader.conf              — systemd-boot configuration
/loader/entries/milnet.conf      — MILNET boot entry with dm-verity root hash
```

**Security properties:**
- Kernel binary is signed for UEFI Secure Boot (MOK or vendor key)
- Firmware verifies bootloader signature before execution
- Bootloader verifies kernel signature before execution
- Root hash in boot entry binds kernel to specific rootfs version
- Partition is mounted read-only after boot (or not mounted at all)

---

### /dev/sda2 — Root Filesystem (dm-verity Protected)

| Property     | Value                          |
|-------------|--------------------------------|
| Size        | 5 GB                           |
| Filesystem  | ext4                           |
| Mount Point | / (read-only)                  |
| Protection  | dm-verity (Merkle hash tree)   |

**Contents:**

```
/sbin/init                       — systemd (PID 1)
/usr/local/bin/milnet-*          — 8 MILNET Rust service binaries
  milnet-auth-service
  milnet-session-service
  milnet-token-service
  milnet-admin-service
  milnet-audit-service
  milnet-mfa-service
  milnet-saml-service
  milnet-scim-service
/usr/bin/postgres                — PostgreSQL 16 server
/usr/lib/postgresql/             — PostgreSQL shared libraries
/etc/systemd/system/             — Service unit files
/etc/nftables.conf               — Firewall rules
/usr/lib/                        — Shared libraries (glibc, openssl, etc.)
```

**Security properties:**
- **Immutable:** The partition is NEVER written to after image creation
- **Verified:** Every 4096-byte block is checked against a SHA-256 Merkle hash tree
  on read. Verification is performed by the kernel's dm-verity subsystem.
- **Tamper-evident:** If any block is modified (even one bit), the kernel returns
  an I/O error for that block. The affected service crashes. This is fail-closed.
- **Offline attack resistant:** An attacker with physical disk access cannot
  modify binaries without invalidating the hash tree, which would require
  knowing the signing key to produce a valid new root hash signature.
- The root hash is passed via kernel command line and verified against a
  PKCS#7 signature (CONFIG_DM_VERITY_VERIFY_ROOTHASH_SIG).
- Mount options: `ro,noatime` (read-only, no access time updates)

**Why 5 GB?**
- MILNET binaries: ~200 MB (8 statically-linked Rust services)
- PostgreSQL: ~100 MB
- Base system (glibc, systemd, coreutils): ~500 MB
- Shared libraries: ~200 MB
- Headroom for updates: ~4 GB
- No desktop environment, no documentation, no man pages, no compilers

---

### /dev/sda3 — Verity Hash Partition

| Property     | Value                          |
|-------------|--------------------------------|
| Size        | 256 MB                         |
| Filesystem  | None (raw dm-verity hash data) |
| Mount Point | None (accessed by kernel only) |
| Protection  | Signed root hash               |

**Contents:**
- Merkle hash tree generated by `veritysetup format`
- Binary structure: tree of SHA-256 hashes, 4096-byte blocks
- Leaf nodes: hash of each 4096-byte data block in /dev/sda2
- Interior nodes: hash of child node hashes
- Root hash: single SHA-256 value at the top of the tree

**Security properties:**
- This partition is written once during image creation and never modified
- The root hash (top of the Merkle tree) is embedded in the kernel command line
- The root hash is signed with PKCS#7 using the MILNET signing key
- The kernel verifies the root hash signature at boot before activating verity
- Size calculation: ~16 bytes of hash per 4096-byte data block, plus tree overhead
  For a 5 GB root: ~1.3 million blocks x ~32 bytes = ~40 MB (256 MB provides headroom)

**Why a separate partition?**
- Separation of data and metadata prevents an attacker from reallocating
  hash space to store a modified rootfs
- Fixed partition boundaries are enforced by GPT, not by the filesystem
- The kernel accesses this partition directly via the block layer

---

### /dev/sda4 — Encrypted Data Partition (LUKS2 + TPM2)

| Property     | Value                          |
|-------------|--------------------------------|
| Size        | Remaining disk space           |
| Outer       | LUKS2 (AES-256-XTS, Argon2id) |
| Inner FS    | ext4                           |
| Mount Point | /var/lib/milnet                |
| Auto-unlock | TPM2 PCR-sealed key            |

**Contents (inside encrypted volume):**

```
/var/lib/milnet/
  config/                        — Service configuration files
    auth-service.toml
    session-service.toml
    ...
  state/                         — Service state (session tokens, caches)
  audit-log/                     — Audit event logs
  tls/                           — TLS certificates and private keys
    server.crt
    server.key                   — (mode 0600, owned by milnet user)
    ca.crt

/var/lib/postgresql/16/main/     — PostgreSQL data directory
  base/                          — Database files
  pg_wal/                        — Write-ahead logs
  pg_xact/                       — Transaction status
  postgresql.conf                — Database configuration
  pg_hba.conf                    — Client authentication
```

**LUKS2 parameters:**
- Cipher: `aes-xts-plain64` (AES-256 in XTS mode)
- Key size: 512 bits (256-bit AES key + 256-bit XTS tweak key)
- Hash: SHA-512 (for key derivation)
- KDF: Argon2id (memory-hard, resists GPU/ASIC brute force)
  - Memory: 1 GB
  - Parallelism: 4 threads
  - Iterations: 4
- Sector size: 4096 bytes

**TPM2 auto-unlock:**
- LUKS master key is sealed to TPM2 persistent handle `0x81000001`
- Sealed to PCR values:
  - **PCR 0:** Platform firmware (UEFI)
  - **PCR 2:** Option ROM code
  - **PCR 4:** Boot manager code (kernel)
  - **PCR 7:** Secure Boot state (keys enrolled in firmware)
- If ANY measured component changes (firmware update, different kernel,
  Secure Boot key change), the TPM refuses to unseal the key
- This binds the encrypted data to the exact boot chain — a compromised
  kernel cannot decrypt the data partition

**Mount options:** `nosuid,nodev,noatime`
- `nosuid`: prevent SUID binaries on the data partition
- `nodev`: prevent device files on the data partition
- `noatime`: no access time updates (performance + wear reduction)

## Partition Creation Commands

```bash
# Partition the disk (GPT)
parted /dev/sda -- mklabel gpt

# EFI System Partition (512 MB)
parted /dev/sda -- mkpart ESP fat32 1MiB 513MiB
parted /dev/sda -- set 1 esp on

# Root filesystem (5 GB, dm-verity data)
parted /dev/sda -- mkpart milnet-root ext4 513MiB 5633MiB

# Verity hash tree (256 MB)
parted /dev/sda -- mkpart milnet-hash 5633MiB 5889MiB

# Encrypted data (remaining space)
parted /dev/sda -- mkpart milnet-data 5889MiB 100%

# Format EFI partition
mkfs.fat -F32 -n MILNET-EFI /dev/sda1

# Root and verity partitions are handled by dm-verity-setup.sh
# Data partition is handled by dm-verity-setup.sh (LUKS2 section)
```

## Boot Chain Trust Hierarchy

```
UEFI Firmware (hardware root of trust)
  |
  ├── Verifies: Secure Boot keys (PK → KEK → db)
  |
  └── Loads: Shim bootloader (signed by Microsoft UEFI CA)
        |
        ├── Verifies: MOK (Machine Owner Key) database
        |
        └── Loads: MILNET kernel (signed by MOK)
              |
              ├── Verifies: dm-verity root hash signature (PKCS#7)
              |
              ├── Activates: dm-verity on /dev/sda2 using /dev/sda3
              |     Every block read is verified against hash tree
              |
              ├── TPM2 unseal: LUKS key (bound to PCRs 0,2,4,7)
              |     Fails if boot chain was modified
              |
              └── Mounts: / (verified, read-only) + /var/lib/milnet (encrypted)
                    |
                    └── Starts: systemd → MILNET services + PostgreSQL
```

Each step in this chain cryptographically verifies the next. Compromising
any single component breaks the chain and the system refuses to boot
(either Secure Boot rejects the binary, dm-verity returns I/O errors,
or the TPM refuses to unseal the LUKS key).

## Disaster Recovery

If the boot chain is intentionally modified (e.g., kernel update):

1. Build new kernel with `build-kernel.sh --sign`
2. Rebuild rootfs image and run `dm-verity-setup.sh` to generate new hash tree
3. Update boot entry with new root hash
4. Re-seal LUKS key to new PCR values:
   ```bash
   tpm2_unseal -c 0x81000001 -p "pcr:sha256:0,2,4,7" > /tmp/luks.key
   # Boot new kernel, then:
   tpm2_evictcontrol -C o -c 0x81000001  # remove old sealed object
   tpm2_createprimary -C o -c primary.ctx
   tpm2_create -C primary.ctx -i /tmp/luks.key -u sealed.pub -r sealed.priv -L "sha256:0,2,4,7"
   tpm2_load -C primary.ctx -u sealed.pub -r sealed.priv -c sealed.ctx
   tpm2_evictcontrol -C o -c sealed.ctx 0x81000001
   shred -vfz -n 10 /tmp/luks.key
   ```
5. Reboot and verify

**Emergency recovery (lost TPM state):**
A backup LUKS key MUST be stored in a physical safe (printed on paper or
stored on an air-gapped USB drive). This is the only recovery path if
the TPM is cleared or the motherboard is replaced.
